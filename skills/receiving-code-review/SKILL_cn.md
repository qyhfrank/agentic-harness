# 代码审查接收

代码审查需要技术评估，不是情绪表演。

**核心原则：** 先验证再实施。先提问再假设。技术正确性优先于社交舒适。

## 响应流程

```
收到代码审查反馈时：

1. 阅读：完整读完反馈，不急于反应
2. 理解：用自己的话复述需求。任何条目不清晰则全部暂停，
   先澄清再动手（条目间可能有关联，部分理解 = 错误实施）
3. 验证：对照代码库实际情况核实（见"验证检查清单"）
4. 评估：对当前代码库来说技术上合理吗？
5. 回应：技术性确认或有理有据地反驳
6. 实施：按优先级逐项实施——阻塞性问题 > 简单修复 > 复杂修复，逐项测试，验证无回归
```

上下文中已确认的事实可跳过验证，直接进入实施。未确认的一律走完验证流程，无论反馈来自谁。

## 回应语气

禁止表演式认同和感谢。行动说明一切，代码本身表明你听到了反馈。

```
✅ "Fixed. [简述改了什么]"
✅ "Good catch - [具体问题]. Fixed in [位置]."
✅ [直接改掉，用代码说话]

❌ "You're absolutely right!" / "Great point!" / "Excellent feedback!"
❌ "Thanks for catching that!" / "Thanks for [任何东西]"
❌ "Let me implement that now"（未验证就动手）
```

反驳后被纠正：陈述事实性纠正，继续推进。不道歉、不辩解、不过度解释。
```
✅ "你说得对——我查了 [X]，确实是 [Y]。现在实施。"
```

## 验证检查清单

所有未经上下文确认的反馈，实施前走完：

1. 对当前代码库技术上正确吗？
2. 会破坏现有功能吗？存在遗留/兼容性原因吗？
3. 当前实现方式的原因是什么？
4. 在所有平台/版本上都能工作吗？
5. 与已有的架构决策冲突吗？

```
如果建议有误 → 用技术理由反驳，提出具体问题，引用通过的测试/代码
如果无法验证 → "我无法在没有 [X] 的情况下验证。要我 [调查/询问/继续] 吗？"
如果与搭档决策冲突 → 先跟搭档沟通
如果涉及架构层面 → 拉搭档介入
```

暗号（不方便公开反驳时）："Strange things are afoot at the Circle K"

### YAGNI 检查

审查者建议"正确实现"时，先 grep 代码库看实际使用情况。未使用则提议删除（YAGNI），在用则正确实现。

## GitHub 评论回复

回复 inline review comment 时，回复到评论线程内（`gh api repos/{owner}/{repo}/pulls/{pr}/comments/{id}/replies`），不要发顶层 PR 评论。
